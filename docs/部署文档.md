# 部署文档（怎么部署到服务器上）

> 这篇教你怎么把服务部署到实际环境中跑。接口怎么调看 [使用文档](使用文档.md)。

---

## 1. 选哪种部署方式

| 方式 | 适合什么情况 | 说明 |
|---|---|---|
| 本地直接跑 | 开发调试、或者服务和你的脚本在同一台机器上 | 最简单，见下面第 2 节 |
| Docker（CPU 版） | 部署到服务器上、没有显卡 | 把环境打包成一个容器，见第 3 节 |
| Docker（GPU 版） | 有 NVIDIA 显卡、想加速识别 | 需要额外装显卡工具，见第 3 节 |

模型文件和模板图**不打包进容器里**，运行时用文件夹挂载的方式传进去，方便更新。

---

## 2. 本地 / 服务器直接跑

```bash
git clone https://gitee.com/xiaozai-van-liu/OcrYoloEngine.git
cd OcrYoloEngine
uv sync --extra yolo --extra ocr          # 装完整依赖
uv run ocr-yolo serve --host 0.0.0.0 --port 8000
```

> 如果指定端口被占用，服务会自动尝试下一个端口（8001、8002……），并在终端打印实际使用的端口。

### 让服务在后台持续运行

上面的命令会在前台运行——关掉终端服务就停了。要让它在后台一直跑，可以用 systemd（Linux 系统自带的服务管理器）。

创建配置文件 `/etc/systemd/system/ocr-yolo.service`：

```ini
[Unit]
Description=OcrYoloEngine 视觉识别服务
After=network.target

[Service]
WorkingDirectory=/opt/OcrYoloEngine
Environment=OYE_DEVICE=cpu
Environment=OYE_API_KEYS=["prod-key-1"]
ExecStart=/home/USER/.local/bin/uv run ocr-yolo serve --host 0.0.0.0 --port 8000
Restart=always

[Install]
WantedBy=multi-user.target
```

然后启用：

```bash
sudo systemctl daemon-reload && sudo systemctl enable --now ocr-yolo
```

这样服务会开机自启，崩溃了也会自动重启。

## 3. 用 Docker 部署

项目的 `docker/` 目录下提供了两套 Docker 配置文件。

### 3.1 CPU 版（无显卡）

```bash
# 在项目根目录构建镜像
docker build -f docker/Dockerfile.cpu -t ocr-yolo:cpu .

# 运行容器：挂载模型/模板/配置目录，映射端口
docker run -d --name ocr-yolo -p 8000:8000 \
  -e OYE_DEVICE=cpu \
  -e OYE_API_KEYS='["prod-key-1"]' \
  -v $(pwd)/models_store:/app/models_store \
  -v $(pwd)/templates_store:/app/templates_store \
  -v $(pwd)/configs:/app/configs \
  ocr-yolo:cpu
```

> 正式部署前，先复制一份实际配置再改（不复制也能跑，会自动回退用自带的 demo 示例）：
> ```bash
> cp configs/models.yaml.example configs/models.yaml
> cp configs/templates.yaml.example configs/templates.yaml
> ```
> 挂载 `configs` 目录后，改宿主机上的 `configs/models.yaml` 即可，容器里实时生效。

### 3.2 GPU 版（有 NVIDIA 显卡）

前提：服务器上要装好 NVIDIA 显卡驱动 + [nvidia-container-toolkit](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html)（让 Docker 能用显卡）。

```bash
docker build -f docker/Dockerfile.gpu -t ocr-yolo:gpu .

docker run -d --name ocr-yolo --gpus all -p 8000:8000 \
  -e OYE_DEVICE=cuda \
  -v $(pwd)/models_store:/app/models_store \
  -v $(pwd)/templates_store:/app/templates_store \
  -v $(pwd)/configs:/app/configs \
  ocr-yolo:gpu
```

> 镜像内用 uv 按锁定的版本安装依赖，跟本地开发用的是同一套工具、同一套版本。首次构建需要下载深度学习框架，比较慢。

### 3.3 用 docker compose（更方便管理）

```yaml
services:
  ocr-yolo:
    build:
      context: .
      dockerfile: docker/Dockerfile.cpu
    ports: ["8000:8000"]
    environment:
      OYE_DEVICE: cpu
      OYE_MAX_WORKERS: "4"
    volumes:
      - ./models_store:/app/models_store
      - ./templates_store:/app/templates_store
      - ./configs:/app/configs
    restart: always
```

## 4. 配置（环境变量）

所有配置都用 `OYE_` 开头的环境变量，或在项目根目录放 `.env` 文件。完整清单见 [使用文档](使用文档.md)。部署时最常调的几个：

| 变量 | 建议 |
|---|---|
| `OYE_DEVICE` | 没显卡用 `cpu`；有显卡用 `cuda` 或 `auto`（自动检测） |
| `OYE_API_KEYS` | **正式环境一定要设**，比如 `["k1","k2"]`，开启密钥验证 |
| `OYE_ALLOWED_PATH_ROOTS` | 如果用本地路径传图，要设成允许读取的目录，比如 `["/data/shots"]` |
| `OYE_MAX_WORKERS` | 同时处理多少个请求，根据 CPU 核数或 GPU 显存来调 |
| `OYE_MAX_QUEUE` | 排队上限，超出就直接拒绝，防止请求堆积把服务拖垮 |
| `OYE_REQUEST_TIMEOUT_S` | 单个请求最长处理多少秒，超时返回 504 |
| `OYE_MODEL_CACHE_SIZE` | 同时在内存里保留几个模型，显存紧张就调小 |

## 5. 健康检查

- `GET /health`：检查服务是否还在运行（活着就返回 200）。
- `GET /ready`：检查服务是否准备好了，会返回已配置的模型列表。

如果用 Kubernetes 部署，可以这样配置探针：

```yaml
livenessProbe:
  httpGet: { path: /health, port: 8000 }
  initialDelaySeconds: 10
readinessProbe:
  httpGet: { path: /ready, port: 8000 }
  initialDelaySeconds: 15
```

## 6. 性能调优建议

- **CPU 部署**：`OYE_MAX_WORKERS` 设成跟 CPU 核数差不多。单次识别通常几十到几百毫秒，能同时处理多少主要取决于 CPU 核数。
- **GPU 部署**：同一个模型的识别会串行执行（因为模型本身不支持多线程同时用）。想提高吞吐量可以减少模型切换、合理设置缓存数量，或者跑多个服务实例做负载均衡。
- **排队限流**：`OYE_MAX_QUEUE` 控制最多排几个请求。满了就直接告诉调用方"太忙了请稍后再试"（503），调用方应该根据这个信号等一下再重试，而不是一直发请求。
- **图片大小限制**：`OYE_MAX_IMAGE_BYTES` 和 `OYE_MAX_IMAGE_PIXELS` 防止超大图把内存撑爆，根据你的业务图片大小设置。
- **多实例扩展**：服务本身不存储状态（模型和模板来自挂载的目录），可以同时跑多个实例，前面放一个负载均衡器（比如 Nginx）分发请求。

## 7. 安全建议

- 正式环境**一定要设密钥**（`OYE_API_KEYS`），不要让任何人都能调。
- 用本地路径传图时，**必须**设 `OYE_ALLOWED_PATH_ROOTS` 限定只能读哪些目录（否则可能被利用读取服务器上的其他文件）。
- 不要把服务直接暴露到公网上，要放在反向代理（如 Nginx）后面，加上 HTTPS 加密。
- 模型文件（`.pt` 等）不在代码仓库里（已被忽略），通过文件夹挂载或镜像构建来分发。

---

## 8. 升级与换模型

- **升级代码**：`git pull` → `uv sync --extra yolo --extra ocr` → 重启服务。Docker 环境则重新构建镜像。
- **换模型不用重启**：更新 `models_store/` 下的模型文件，改好 `configs/models.yaml` 里的版本号，然后调 `POST /v1/models/{name}/reload` 就行。
- **依赖版本锁定**：`uv.lock` 文件记录了精确的依赖版本，保证在任何机器上装出来的都一样。
