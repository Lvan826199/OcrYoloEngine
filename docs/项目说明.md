## 1. 一句话介绍

`OcrYoloEngine` 是一个**帮自动化脚本"看懂"屏幕截图**的服务。

简单说：**你的自动化脚本截了一张图，发给这个服务，服务告诉你"图里的某个按钮/文字在什么位置"，你的脚本再根据这个位置去点击**。

本服务**只负责"看"**——告诉你东西在哪。**不负责"点"**——点击、滑动等操作由你的脚本自己做。

用在哪些场景：

- 手机 App 截图里**找文字**（比如"登录"按钮在哪个坐标）
- 网页截图里**找元素**
- 手机**游戏**截图里找图标、按钮、角色等

---

## 2. 三种"看图"方式（互相配合）

| 方式 | 用什么技术 | 擅长干什么 | 返回什么 |
|---|---|---|---|
| **文字识别（OCR）** | PaddleOCR | 找出图里的**文字**在哪 | 识别出的文字 + 位置框 + 把握程度 |
| **目标检测（YOLO）** | ultralytics（v8/v11） | **找特定物体**（如游戏里的 boss、金币），需要提前训练模型 | 物体类别 + 位置框 + 把握程度 |
| **模板匹配** | OpenCV（自动缩放 + 去重） | 找**固定不变的小图标**（类似"按图索骥"） | 模板名 + 位置框 + 相似程度 |

三种方式**用同一套接口、返回同一种格式**的结果：你不管用哪种方式，拿到的数据结构都一样——坐标、文字、把握程度，拿来就能用。

---

## 3. 使用流程（你作为调用方怎么用）

```
1. 把这个服务跑起来（本地跑或者用 Docker 部署到服务器）
2. 在 configs/ 目录里登记好你的模型和模板图（文字识别不用，自带模型）
3. 你的自动化脚本截图 → 得到一张图片
4. 把图片发给服务的接口（可以传图片编码、本地路径、或直接上传文件）
5. 拿到结果：每个识别到的目标的位置框、中心点坐标、文字内容、把握程度
6. 你的脚本拿着坐标去点击或做断言
```

具体怎么调接口、每个字段什么意思，看 [使用文档](使用文档.md)。

## 4. 一次请求在服务内部经历了什么

以"用 YOLO 找物体"为例：

```
你发了一个请求过来（带着图片 + 要用哪个模型 + 其他参数）
  → 先检查身份（如果开了密钥验证的话）
    → 读取图片（如果是本地路径，还要检查路径是否在允许范围内）
      → 图片预处理（检查大小、统一颜色格式、如果指定了区域就裁剪）
        → 排队等工作线程（如果队列满了直接告诉你"太忙了请稍后再试"）
          → 加载模型（第一次用才加载，用过的会缓存起来）→ 开始识别
            → 把识别结果的坐标换算回原图的位置
              → 打包成标准格式返回给你
```

有一个重要设计：**三种识别方式都只在"处理过的小图"上干活，坐标换算统一由预处理层完成**——这样不管你用哪种方式，返回的坐标口径都是一致的。

## 5. 服务的层次结构

```
            ┌──────────────────────────────────────────┐
 你的请求 → │ 网络层（接收请求、验证身份、分发路由）      │
            └──────────────┬───────────────────────────┘
                           ▼
            ┌──────────────────────────────────────────┐
            │ 排队层（控制同时处理多少个请求，防止过载）   │
            └──────────────┬───────────────────────────┘
                           ▼
 读取图片 → 预处理（统一格式 / 裁剪指定区域 / 坐标换算）
                           ▼
            ┌─────────── 三种识别方式 ───────────────┐
            │  文字识别      目标检测       模板匹配    │
            │ (PaddleOCR)  (ultralytics)  (OpenCV)   │
            └──────┬──────────┬───────────────┬──────┘
                   ▼          ▼               ▼
            模型管理仓库            模板图管理仓库
          （按需加载/版本管理）    （加载缓存/版本管理）
```

各部分说明（对应 `src/ocr_yolo_engine/` 下的文件）：

| 模块 | 干什么 |
|---|---|
| `settings.py` | 集中管理所有配置项（通过环境变量 `OYE_*` 设置） |
| `schemas.py` | 定义请求和响应的数据格式 |
| `errors.py` | 统一的错误码和错误提示 |
| `image/loader.py` | 读取图片：支持编码字符串 / 本地文件路径（有安全限制） / 文件上传 |
| `preprocessing/pipeline.py` | 图片预处理：格式统一、大小检查、区域裁剪、坐标换算 |
| `recognizers/` | 三种识别器的实现代码 |
| `models/registry.py` | 模型管理：按需加载 + 版本追踪 + 自动释放不常用的（最多缓存几个） |
| `templates/store.py` | 模板图管理：加载缓存 + 版本追踪 |
| `concurrency/executor.py` | 排队和并发控制（防止同时太多请求把服务压垮） |
| `observability/logging.py` | 日志记录，每个请求有唯一编号方便排查 |
| `service/` | 网络服务相关：启动应用 / 接口路由 / 身份验证 |
| `cli.py` | 命令行工具：`serve` 启动服务 / `infer` 本地测试单张图 |
| `training/` | **独立的**模型训练脚本（跟服务代码完全隔离，服务运行时不会用到） |

## 6. 核心设计要点（大白话版）

- **按需加载，不浪费资源**：文字识别和目标检测的底层库很大（几百 MB），只有你真正用到对应功能时才会加载。如果你只用模板匹配，那些大库完全不会被触碰，启动很快。
- **多模型切换**：可以在配置文件里登记多个模型，请求时指定用哪个。用过的模型会缓存在内存里，超过数量上限就自动把最久没用的释放掉，防止内存爆掉。
- **CPU 和 GPU 都能跑**：默认自动检测——有显卡就用显卡加速，没有就用 CPU。通过配置可以强制指定。
- **不会被压垮**：识别是很吃性能的操作，服务用了一个"排队 + 限流"的机制——同时处理的请求有上限，超出的会被直接拒绝（告诉你"太忙了，等会再试"），不会无限堆积把服务拖死。
- **坐标统一**：返回的坐标有两种格式——**像素坐标**（直接用来点击）和**比例坐标**（0~1 之间的值，不同分辨率的设备通用）。
- **没找到不是错误**：如果图里没有你要找的东西，服务正常返回 200，只是结果列表为空。只有真出问题（图片损坏、模型不存在等）才报错。
- **安全防护**：用本地路径传图时，路径必须在你配置的"允许目录"里（防止被恶意读取服务器上的其他文件）；图片大小和分辨率有上限（防止超大图把内存撑爆）；可以设置密钥验证，防止未授权访问。
- **运行状态可观察**：每个请求都有唯一编号，日志里可以追踪；响应里带每种识别方式的耗时和版本信息；`/metrics` 接口可以接入监控系统查看请求量和耗时统计；`debug=true` 返回标注图——在原图上画框标出识别结果，方便排查"为什么没找到"。
- **不重启换模型**：更新了模型文件后，调一个接口就能让新模型生效，不用重启整个服务。
- **测试扎实**：所有测试都用**真实图片和真实识别器**，不用假数据糊弄。124 个测试全部通过。
