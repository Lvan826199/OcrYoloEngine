## 1. 身份验证（密钥）

- 默认**不需要密钥**（方便本地开发测试）。
- 如果设置了环境变量 `OYE_API_KEYS`（比如 `OYE_API_KEYS='["my-secret-key"]'`），那所有 `/v1/*` 开头的接口都需要在请求头里带上密钥：

```
X-API-Key: my-secret-key
```

没带或者带错了会返回 `401`（未授权）。`/health`、`/ready`、`/metrics` 这几个接口不需要密钥。

## 2. 图片怎么传给服务（三种方式，选一种就行）

| 方式 | 怎么用 | 说明 |
|---|---|---|
| 图片编码 | 请求 JSON 里的 `image.base64` 字段 | 把图片文件转成 base64 编码的字符串传过来 |
| 本地路径 | 请求 JSON 里的 `image.path` 字段 | 直接告诉服务图片在服务器上的路径（只能读允许的目录里的文件） |
| 文件上传 | `POST /v1/recognize/upload` | 用表单的方式上传图片文件 |

JSON 接口里 `image` 必须**且只能**用"编码"或"路径"其中一种，不能两个都写。

## 3. 所有接口一览

| 请求方式 | 地址 | 作用 | 说明 |
|---|---|---|---|
| POST | `/v1/ocr` | 只做文字识别 | 请求体只需要传图片 |
| POST | `/v1/detect` | 只做目标检测 | 需要指定用哪个模型（`model` 字段） |
| POST | `/v1/match` | 只做模板匹配 | 需要指定用哪些模板（`templates` 字段） |
| POST | `/v1/recognize` | 一次跑多种方式 | 可以同时跑文字识别+目标检测+模板匹配 |
| POST | `/v1/recognize/upload` | 文件上传版 | 跟 recognize 一样，但图片用上传方式传 |
| GET | `/v1/models` | 查看已登记的模型 | 返回模型名和版本 |
| POST | `/v1/models/{name}/unload` | 释放模型内存 | 把某个模型从内存里卸掉，释放资源 |
| POST | `/v1/models/{name}/reload` | 重新加载模型 | 更新了模型文件后用这个让新版生效 |
| GET | `/v1/templates` | 查看已登记的模板 | 返回模板名和版本 |
| GET | `/health` | 检查服务是否还活着 | 返回 `{"status":"ok"}` |
| GET | `/ready` | 检查服务是否准备好了 | 返回已配置的模型列表 |
| GET | `/metrics` | 查看运行统计 | 返回各识别方式的请求次数和耗时统计 |

> ⚠️ **格式区别**：`/v1/ocr` 的请求体直接就是图片 `{"base64": "..."}`，不需要外面套 `image`；而 `/v1/detect`、`/v1/match`、`/v1/recognize` 的请求体需要把图片放在 `image` 字段里 `{"image": {"base64": "..."}, "methods": [...], ...}`。如果你从 `/v1/ocr` 换到其他接口时报 422 错误，多半是请求格式没改对。

### 3.1 不重启更换模型

有两个管理接口，可以在不重启服务的情况下切换或刷新模型：

- `POST /v1/models/{name}/unload`：把指定模型从内存里卸掉，**释放内存/显存**。模型名必须是在 `configs/models.yaml` 里登记过的，否则返回 404。如果这个模型本来就没加载，啥也不会发生（不会报错）。
- `POST /v1/models/{name}/reload`：**重新加载模型**——先把旧的卸掉，再重新加载。用法：你更新了 `models_store/` 下的模型文件，改好 `configs/models.yaml` 里的版本号，然后调一次 reload 就生效了。

```bash
# 释放某个模型的内存
curl -X POST http://127.0.0.1:8000/v1/models/game/unload -H "X-API-Key: <密钥>"
# 让新版模型生效
curl -X POST http://127.0.0.1:8000/v1/models/game/reload -H "X-API-Key: <密钥>"
```

> reload 会真正加载模型文件，如果模型比较大可能需要几秒到十几秒，这是正常的。

## 4. 请求格式说明

下面是完整请求的所有字段（只有 `image` 和 `methods` 是必填的，其余都可选）：

```jsonc
{
  "image": { "base64": "..." },   // 或者 { "path": "/允许的目录/截图.png" }
  "methods": ["yolo"],            // 用哪种方式：可选 "ocr"、"yolo"、"template"，可以多选
  "model": "game_a",              // 用目标检测时必填：指定用哪个模型
  "templates": ["settings_icon"], // 用模板匹配时必填：指定用哪些模板
  "conf_threshold": 0.4,          // 可选，0~1，只返回把握程度高于这个值的结果
  "roi": { "x": 100, "y": 200, "w": 300, "h": 400 }, // 可选，只在图片的这个区域内识别
  "debug": false,                 // 可选，设为 true 会额外返回一张标注了结果的图
  "merge": "none",                // 可选，多方式合并策略，下面有详细说明
  "cache": "auto"                 // 可选，缓存行为（auto/refresh/off）
}
```

几个注意点：

- 用了 `"yolo"` 但没给 `model`，或者用了 `"template"` 但没给 `templates` → 返回 `422`（请求格式不对）。
- `roi`（感兴趣区域）：只在图片的指定矩形范围内识别，但返回的坐标会**自动换算回原图坐标**，不用你自己加偏移。
- `merge`：多种方式一起用时的合并策略（只在 `/v1/recognize` 生效），详见下面第 7 节。

## 5. 返回结果格式说明

```jsonc
{
  "request_id": "9f2c...",          // 本次请求的唯一编号，排查问题时有用
  "image_size": [1080, 1920],       // 原图的 [宽, 高]（像素）
  "method_results": {
    "yolo": {
      "detections": [
        {
          "source": "yolo",            // 这个结果来自哪种识别方式
          "label": "boss",             // 目标检测的类别名 / 模板名；文字识别这里是空的
          "text": null,                // 文字识别的文字内容；其他方式这里是空的
          "confidence": 0.93,          // 把握程度，0~1，越大越有把握
          "bbox": [120, 340, 260, 520],      // 目标框：[左上角x, 左上角y, 右下角x, 右下角y]
          "center": [190, 430],              // 目标中心点：[x, y]——直接用来点击
          "bbox_norm": [0.111, 0.177, 0.240, 0.270], // 按比例换算的目标框（0~1）
          "center_norm": [0.175, 0.223]               // 按比例换算的中心点（0~1）
        }
      ],
      "model_version": "v1",          // 用的模型版本号
      "template_versions": null,      // 模板匹配时这里会有各模板的版本
      "elapsed_ms": 18.4              // 这种方式花了多少毫秒
    }
  },
  "debug_image": null,              // debug=true 时这里是标注图的编码
  "merged": null,                   // 使用合并策略时这里是合并后的结果列表
  "from_cache": false               // 是否来自缓存
}
```

**坐标怎么用**：
- `bbox = [x1, y1, x2, y2]`：目标的左上角和右下角坐标（像素），画框用。
- `center = [cx, cy]`：目标的中心点坐标——**你的脚本一般直接点这个坐标就行**。
- `*_norm` 结尾的是"按比例换算"的版本（值在 0~1 之间），不同分辨率的设备之间通用。

**没找到目标** → 返回 `200`（正常），对应方式的 `detections` 是空数组 `[]`，**这不是报错**。

## 6. 调用示例

### 6.1 文字识别（最简单，不用准备模型）

```bash
# 把截图转成编码字符串，发给文字识别接口
B64=$(base64 -w0 screenshot.png)
curl -s http://localhost:8000/v1/ocr \
  -H "Content-Type: application/json" \
  -d "{\"base64\": \"$B64\"}"
```

### 6.2 目标检测（需要指定模型）

```bash
curl -s http://localhost:8000/v1/detect \
  -H "Content-Type: application/json" \
  -d '{"image": {"path": "/data/shots/a.png"}, "methods": ["yolo"], "model": "game_a", "conf_threshold": 0.5}'
```

### 6.3 模板匹配

```bash
curl -s http://localhost:8000/v1/match \
  -H "Content-Type: application/json" \
  -d '{"image": {"base64": "..."}, "methods": ["template"], "templates": ["settings_icon"]}'
```

### 6.4 一次跑多种方式

```bash
curl -s http://localhost:8000/v1/recognize \
  -H "Content-Type: application/json" \
  -d '{"image": {"base64": "..."}, "methods": ["ocr", "yolo"], "model": "game_a"}'
# 返回里 method_results 会同时有 "ocr" 和 "yolo" 两组结果
```

### 6.5 用文件上传的方式

```bash
curl -s http://localhost:8000/v1/recognize/upload \
  -F "file=@screenshot.png" \
  -F "methods=ocr,template" \
  -F "templates=settings_icon"
```

### 6.6 带密钥

```bash
curl -s http://localhost:8000/v1/ocr \
  -H "X-API-Key: my-secret-key" \
  -H "Content-Type: application/json" \
  -d '{"base64": "..."}'
```

### 6.7 用 Python 代码调用

```python
import base64, requests

# 读取截图并转成编码
with open("screenshot.png", "rb") as f:
    b64 = base64.b64encode(f.read()).decode()

# 发请求
resp = requests.post(
    "http://localhost:8000/v1/recognize",
    json={"image": {"base64": b64}, "methods": ["ocr", "yolo"], "model": "game_a"},
    headers={"X-API-Key": "my-secret-key"},  # 没开密钥验证可以不写这行
)

# 解析结果
data = resp.json()
for det in data["method_results"].get("yolo", {}).get("detections", []):
    cx, cy = det["center"]
    print(f"{det['label']} 把握 {det['confidence']:.2f} → 点击坐标 ({cx}, {cy})")
```

## 7. 多方式合并（只在 /v1/recognize 生效）

`/v1/recognize` 可以一次跑多种识别方式。默认各方式的结果是分开的（在 `method_results` 里各自一组）。如果你想要一份**合并后的统一列表**，可以用 `merge` 字段指定合并策略。

> 注意：单方式接口（`/v1/ocr`、`/v1/detect`、`/v1/match`）不受影响，始终各自返回。

| 策略 | 怎么合并 | `merged` 字段 |
|------|----------|--------------|
| `none`（默认） | 不合并，各方式分开返回 | 空（`null`） |
| `priority` | 按你写的顺序逐个试，**一有结果就停**（后面的方式不跑了，省时间） | 第一个有结果的方式的检测列表 |
| `dedup` | 把所有方式的结果汇总，**去掉重复的**（两个框重叠超过 50% 就算同一个，只留把握程度高的） | 去重后的列表 |
| `concat` | 把所有方式的结果简单合在一起，按把握程度从高到低排序 | 合并后的完整列表 |

- `priority` 适合"有 A 就用 A，A 没找到再试 B"的场景，还能节省后面方式的计算时间。
- `dedup` 适合多种方式可能找到同一个目标、你想要去重结果的场景。
- `concat` 适合你只想把所有结果汇成一个列表排序。
- 不管用哪种策略，`method_results` 里仍然保留各方式的原始结果。

示例：

```bash
# concat：汇总所有结果
curl -s http://localhost:8000/v1/recognize \
  -H "Content-Type: application/json" \
  -d '{"image": {"base64": "..."}, "methods": ["template", "ocr"], "templates": ["settings_icon"], "merge": "concat"}'

# priority：有结果就停
curl -s http://localhost:8000/v1/recognize \
  -H "Content-Type: application/json" \
  -d '{"image": {"base64": "..."}, "methods": ["yolo", "template"], "model": "game_a", "templates": ["settings_icon"], "merge": "priority"}'
```

## 8. 命令行工具

```bash
# 启动服务
uv run ocr-yolo serve --host 0.0.0.0 --port 8000

# 本地测试单张图（直接打印 JSON 结果，不用先启动服务）
uv run ocr-yolo infer screenshot.png --methods ocr
uv run ocr-yolo infer a.png --methods yolo --model game_a --conf 0.5
uv run ocr-yolo infer a.png --methods template --templates settings_icon
```

`infer` 命令会自动把图片所在目录设为允许读取的目录，不用手动配置。

## 9. 配置项（环境变量）

所有配置都通过 `OYE_` 开头的环境变量设置，也可以在项目根目录放一个 `.env` 文件（每行一条 `KEY=VALUE`）。

| 变量名 | 默认值 | 说明 |
|---|---|---|
| `OYE_DEVICE` | `auto` | 用什么设备跑：`auto`（自动检测）/ `cpu` / `cuda`（显卡） |
| `OYE_DEFAULT_CONF_THRESHOLD` | `0.25` | 目标检测的默认把握程度门槛（请求里可以单独覆盖） |
| `OYE_MODELS_CONFIG_PATH` | `configs/models.yaml` | 模型配置文件的路径 |
| `OYE_TEMPLATES_CONFIG_PATH` | `configs/templates.yaml` | 模板配置文件的路径 |
| `OYE_MODEL_CACHE_SIZE` | `3` | 最多同时在内存里保留几个模型（超出的自动释放最久没用的） |
| `OYE_RESULT_CACHE_SIZE` | `0` | 结果缓存容量（条数）；**`0` 表示关闭**（默认）——同一张图多次请求不走缓存 |
| `OYE_RESULT_CACHE_TTL_S` | `0` | 缓存的结果多久过期（秒）；`0` 表示不过期 |
| `OYE_MAX_WORKERS` | `4` | 同时处理多少个请求 |
| `OYE_MAX_QUEUE` | `32` | 排队上限，超出直接返回"太忙了"（503） |
| `OYE_REQUEST_TIMEOUT_S` | `30` | 单个请求最长处理多少秒，超时返回 504 |
| `OYE_MAX_IMAGE_BYTES` | `10485760`（10MB） | 输入图片最大允许多少字节 |
| `OYE_MAX_IMAGE_PIXELS` | `16777216`（4096²） | 输入图片最大允许多少像素 |
| `OYE_ALLOWED_PATH_ROOTS` | `[]` | 允许通过本地路径读取图片的目录列表 |
| `OYE_API_KEYS` | `[]` | 允许的密钥列表，空表示不开启密钥验证 |
| `OYE_WARMUP` | `true` | 启动时是否预热 |

> 列表和布尔值用 JSON 格式写：`OYE_API_KEYS='["key1","key2"]'`、`OYE_WARMUP=false`。

### 9.1 配置文件怎么用（重要）

模型和模板的配置分两份：

- `configs/models.yaml.example`、`configs/templates.yaml.example`：**入库的示例模板**，含开箱即用的 demo（`yolov8n` / `demo_block`）。
- `configs/models.yaml`、`configs/templates.yaml`：**你自己的实际配置**，已被 git 忽略，改它不会和上游冲突、也不会误提交。

```bash
# 要配自己的模型/模板时，先复制一份再改：
cp configs/models.yaml.example configs/models.yaml
cp configs/templates.yaml.example configs/templates.yaml
```

> 还没复制也能跑——服务会**自动回退**读取 `.example`，所以装完就能用自带的 demo。一旦你创建了实际的 `configs/models.yaml`，服务就只读它、不再读 `.example`。

### 9.2 登记模型：`configs/models.yaml`

```yaml
models:
  game_a:                          # 模型名（请求里的 model 字段写这个名字）
    path: models_store/game_a.pt   # 模型文件路径（文件需要自己准备，不在代码仓库里）
    version: v1
    classes:                       # 这个模型能识别的类别
      0: boss                      # 编号 0 对应的类别叫 boss
      1: coin                      # 编号 1 对应的类别叫 coin
```

### 9.3 登记模板：`configs/templates.yaml`

```yaml
templates:
  settings_icon:                       # 模板名（请求里的 templates 字段写这个名字）
    path: templates_store/settings.png # 模板图片路径
    version: v1
    params:
      threshold: 0.85                  # 相似度门槛（可选，0~1）
```

### 9.3 结果缓存（默认关闭）

如果同一张图、同一套参数会在短时间内被多次发送（比如轮询、重试），可以开启结果缓存，直接返回上次的结果，省去重复计算。

- **默认关闭**：`OYE_RESULT_CACHE_SIZE=0` 时完全不影响正常使用，连多余的计算都不会做。
- **开启方法**：设 `OYE_RESULT_CACHE_SIZE=N`（比如 `=100`，表示最多缓存 100 条），超出上限会自动丢掉最久没用的；还可以设 `OYE_RESULT_CACHE_TTL_S=T` 让缓存结果在 T 秒后过期。

每个请求可以单独控制缓存行为（通过 `cache` 字段）：

| 模式 | 行为 |
|---|---|
| `auto`（默认） | 先查缓存，有就直接返回；没有就正常计算，算完存进缓存 |
| `refresh` | 不管缓存里有没有，强制重新计算，算完把新结果存进缓存 |
| `off` | 完全不用缓存，既不读也不存 |

返回结果里会告诉你是否用了缓存：
- `from_cache: true` + 响应头 `X-Cache: HIT` = 命中了缓存
- `from_cache: false` + `X-Cache: MISS` = 没命中，这次是新算的
- `X-Cache: BYPASS` = 缓存没开，或者你指定了 `cache=off`

> 当你用"不重启换模型"的接口（unload/reload）更新模型后，缓存会自动清空，防止返回旧结果。

## 10. 错误码速查

出错时返回的格式：

```json
{ "request_id": "...", "error_code": "INVALID_IMAGE", "message": "无法解码为图片", "details": {} }
```

| HTTP 状态码 | 错误码 | 什么意思 |
|---|---|---|
| 400 | `INVALID_IMAGE` | 图片有问题（无法解码 / 编码不合法） |
| 413 | `IMAGE_TOO_LARGE` | 图片太大了（超过字节或分辨率上限） |
| 403 | `PATH_NOT_ALLOWED` | 本地路径不在允许的目录里 |
| 404 | `MODEL_NOT_FOUND` | 指定的模型名没登记过 |
| 404 | `TEMPLATE_NOT_FOUND` | 指定的模板名没登记过 |
| 503 | `OVERLOADED` | 服务太忙了，带 `Retry-After` 头告诉你多久后可以重试 |
| 504 | `TIMEOUT` | 请求处理超时了 |
| 500 | `INTERNAL` | 服务内部出错了 |
| 422 | — | 请求格式有问题（比如用了目标检测但没指定模型名） |

## 11. 常见问题

- **返回的 `detections` 是空的？** 这不是错误——只是没找到达到把握程度门槛的目标。可以把 `conf_threshold` 调低一点再试。
- **图片在同一台机器上，不想每次都传？** 把截图目录加到 `OYE_ALLOWED_PATH_ROOTS` 里，然后请求用 `{"path": "/你的目录/截图.png"}` 就行了。
- **报找不到 torch/paddle？** 文字识别和目标检测的底层库需要额外安装：`uv sync --extra yolo --extra ocr`（详见 [部署文档](部署文档.md)）。
