## 1. 项目目标

`OcrYoloEngine` 是一个给**自动化测试用的"看图"服务**。别的自动化脚本截了图，发给这个服务，服务看完告诉它"图里的东西在哪、文字是什么、有多大把握"，**不负责点击等操作**。

适用场景：

- 手机 App 截图里找文字和按钮位置
- 网页截图里找页面元素
- 手机游戏截图里找图标、角色、道具等

三种识别方式，互相配合：

- **文字识别（OCR）**：用 PaddleOCR 找出图里的文字。
- **目标检测（YOLO）**：用 ultralytics（v8/v11）找特定物体，可以针对不同游戏训练专用模型。
- **模板匹配**：用 OpenCV 按图索骥，适合找固定不变的图标。

### 设计目标

1. 三种方式用同一套接口、返回同一种格式，调用方拿到就能用。
2. 服务**只管识别**；训练是独立的入口，完全隔离开。
3. 多个模型**按需加载、可以切换**，带版本管理和自动释放。
4. **有没有显卡都能跑**，有显卡就自动加速。
5. 图片支持编码传输、文件上传、或者直接传本地路径。
6. 从一开始就考虑并发控制、可测试、可监控、安全等方面。

## 2. 范围

### 首版做了什么

- 网页接口服务（`/v1` 路由）
- 三种识别器 + 统一接口
- 模型管理（按需加载、版本追踪、自动释放、热更新）+ 模板库
- 图片预处理（格式统一、区域裁剪、坐标换算、大小限制）
- 排队限流（防过载 503 / 超时 504）
- 统一的错误格式、日志追踪、运行指标监控
- CPU/GPU 配置 + 按需加载底层库 + Docker 镜像
- 真实数据测试（不用假数据）
- 代码质量检查（格式、类型、测试 + 提交前自动检查）
- 独立的模型训练入口
- 密钥验证
- 结果缓存（可选，默认关闭）
- 多方式合并（priority/dedup/concat）
- 调试标注图（debug=true 返回画框的图）
- Prometheus 监控指标端点

### 暂时不做（但留了扩展空间）

- 不执行任何界面操作（点击、滑动等）
- 不做自动训练调参
- 不做多机集群（先做好单机多线程）

## 3. 整体架构

采用**分层架构**——一个服务进程，各模块各司其职、互不干扰。

```
                ┌──────────────────────────────────────────┐
   你的请求 ──▶ │ 网络层（接收请求、验证身份、分发路由）      │
                └──────────────┬───────────────────────────┘
                               ▼
                ┌──────────────────────────────────────────┐
                │ 排队层（控制并发、防止过载）               │
                └──────────────┬───────────────────────────┘
                               ▼
   读图片 ──▶ 预处理（格式统一 / 区域裁剪 / 坐标换算）
                               ▼
                ┌─────────── 三种识别方式 ──────────────┐
                │  文字识别      目标检测      模板匹配    │
                │ (PaddleOCR)  (ultralytics)  (OpenCV)  │
                └──────┬──────────┬──────────────┬──────┘
                       ▼          ▼              ▼
                模型管理仓库            模板图管理仓库
              （按需加载/版本/自动释放） （加载/版本/缓存）
```

## 5. 各模块说明

### 5.1 配置管理

集中读取环境变量和默认值，所有配置一目了然。完整配置项见 [使用文档](使用文档.md)。

### 5.2 数据格式

**请求**：传图片 + 选方式 + 选模型/模板 + 可选参数（把握程度门槛、区域裁剪、是否返回标注图、合并策略、缓存行为）。

**返回结果**：每个识别到的目标都包含：
- 来源（哪种方式找到的）
- 类别/模板名 或 识别出的文字
- 把握程度（0~1）
- 位置框和中心点坐标（像素 + 按比例换算的两种格式）

### 5.3 图片读取和预处理

- **三种传图方式**：编码字符串 / 本地路径（有安全限制，防恶意读取文件） / 文件上传。
- **预处理**：检查大小、统一颜色格式、按指定区域裁剪。
- **坐标换算**：识别器只在处理后的图上工作，坐标换算回原图**统一由预处理层完成**——三种方式的坐标口径绝对一致。

### 5.4 三种识别器

都遵循同一个接口规范：接收图片、返回结果列表。

- **文字识别**：用 PaddleOCR，支持中英文和数字。
- **目标检测**：用 ultralytics，从模型仓库取指定模型。
- **模板匹配**：用 OpenCV 的模板匹配功能，支持自动缩放适配不同大小、找多个目标、去掉重复框。

### 5.5 资源管理

- **模型管理**：从配置文件读取模型信息，第一次用到才加载，用过的缓存起来，超过上限自动释放最久没用的。支持不重启换模型（卸载 + 重载接口）。
- **模板管理**：类似模型管理，加载后缓存、带版本追踪。

### 5.6 并发控制

识别是很吃 CPU/GPU 的操作，不能无限制地同时跑。服务使用了：
- **排队 + 限流**：超出上限直接拒绝（503），不会无限堆积。
- **同模型串行**：同一个模型同时只有一个线程在用，避免冲突。
- **单请求超时**：超过时间限制返回 504。

### 5.7 错误处理

统一格式：成功有目标 → 200 + 结果；成功没目标 → 200 + 空列表（**不是错误**）；真出问题 → 对应的错误码。

### 5.8 网络服务

接口路由在 `/v1` 下，支持密钥验证（可关闭）。各模块通过组装器注入，方便测试时替换为真实组件做端到端验证。

### 5.9 命令行工具

`ocr-yolo serve` 启动服务；`ocr-yolo infer` 本地测试单张图。底层大库只在真正需要时才加载，不影响轻量命令的速度。

### 5.10 训练入口

完全独立的目录，服务代码不会引用它。训练产出的模型文件放到 `models_store/`，在配置文件里登记就能被服务使用。

## 6. 请求处理流程（以目标检测为例）

```
你的脚本截了图，发过来
  → 验证密钥
    → 读取图片，检查大小和路径权限
      → 预处理：统一格式 + 裁剪指定区域
        → 排队等处理（队列满了就直接拒绝）
          → 从仓库取模型（首次用才加载）→ 开始识别
            → 坐标换算回原图 + 计算比例坐标
              → 打包结果（可选：画标注框、存缓存）
                → 返回 JSON 结果
```

## 7. 测试策略

所有测试都用**真实数据**，不用假的替代品。

- **单元测试**：用真实的图片（代码生成的）和真实的函数来测各个模块。
- **接口测试**：用真实的模板识别器做端到端验证，不是假的。
- **标准样例测试**：仓库里保存了固定的样例图和期望结果，防止改代码时悄悄改变了识别行为。
- **真实模型冒烟测试**：加载真实的 PaddleOCR 和 YOLO 模型跑端到端，但因为需要下载大模型所以单独触发。

---

## 8. 安全

- 本地路径传图：路径必须在允许的目录里（防止被利用读取其他文件）。
- 图片大小和分辨率有上限（防止超大图把内存撑爆）。
- 可选密钥验证。

---

## 9. 可监控

- 每个请求都有唯一编号，日志里可以追踪。
- 返回结果里带每种方式的耗时和版本信息。
- `/metrics` 接口可以接入 Prometheus 等监控系统。
- `debug=true` 返回标注图，方便排查"为什么没找到"。

## ADR 0002 - 模板匹配的相似度计算方式

- 状态：已接受
- 日期：2026-06-03

### 问题

模板匹配需要给每个匹配位置打一个"有多像"的分数（0~1）。最初用的方式在遇到纯色模板（比如纯色按钮图标）时会出问题——所有纯色区域都会被误判为"完全匹配"，产生大量错误结果。而手机游戏里纯色按钮非常常见。

### 选择

采用**像素差异**的方式计算相似度：逐个像素比较差异，完全一样得 1.0，差异越大分数越低。这个方法对纯色模板也稳定可靠，分数含义直观。

代价是对光照变化的适应性稍弱，但对我们的截图场景（截图光照稳定）足够用了。

## ADR 0003 - 结果缓存（可选，默认关闭）

- 状态：已接受
- 日期：2026-06-04

### 问题

同一张图同样的参数可能被短时间内重复发送（轮询、重试等），重复识别浪费算力。但缓存**不能改变默认行为**——不开缓存时必须跟以前完全一样。

### 选择

采用**可选开关 + 默认关闭**的方案：

- 默认关闭（`OYE_RESULT_CACHE_SIZE=0`），连多余的计算都不做，完全不影响现有行为。
- 设了容量就开启，超出上限自动丢掉最久没用的，可选过期时间。
- 每个请求可以单独控制是否用缓存。
- 更换模型后缓存自动清空，防止返回旧结果。

## ADR 0004 - 多方式合并策略

- 状态：已接受
- 日期：2026-06-04

### 问题

`/v1/recognize` 可以一次跑多种方式，但结果是分开的。调用方经常需要一份合并后的统一列表（级联取首个、去重、简单汇总），自己合并容易出错。但**默认行为不能改**。

### 选择

采用**可选策略 + 默认不合并**的方案：

- 默认 `none`：跟以前完全一样。
- `priority`：按顺序试，有结果就停（省后面方式的算力）。
- `dedup`：汇总后去掉重复框（两个框重叠超过 50% 只留把握程度高的）。
- `concat`：简单合在一起按把握程度排序。
- 跟缓存兼容：不同策略的缓存互不影响。
